# Методология расчёта FLOPs

## Общий принцип

Для всех стратегий используется единая формула:

```
C = k × N × D
```

где:
- `N` — число параметров модели
- `D` — суммарное число обработанных токенов
- `k` — множитель, отражающий число операций на параметр на токен

Структура формулы совместима с Kaplan et al. [1] и верифицирована по Snell et al. [4]:
> *"To determine pretraining FLOPs, we use the common approximation X = 6ND_pretrain, and for inference FLOPs, we use Y = 2ND_inference."*

---

## Стратегии

### SFT (`k = 6`)

```
C_SFT = 6 × N × D_train
D_train = total_tokens × n_epochs
```

- `total_tokens` — суммарное число токенов обучающих последовательностей (промпт + ответ), из `data_stats.json`
- `n_epochs = 3`
- `k = 6`: прямой проход (2) + обратный проход (4) на параметр на токен
- LoRA экономит память, но **не** FLOPs: прямой и обратный проходы идут через все `N` параметров

**Источник**: Kaplan et al. (2020), arXiv:2001.08361, формула C ≈ 6ND

---

### GRPO (`k = 10`)

```
C_GRPO = 10 × N × D_GRPO
D_GRPO = num_tokens (из training_log.json, поле num_tokens на шаге t)
```

- `D_GRPO` — суммарное число токенов генераций за всё обучение GRPO, регистрируемое TRL в поле `num_tokens` лога обучения
- Для чекпоинта на шаге `t` используется значение `num_tokens` на этом шаге
- `k = 10`: генерация (2) + прямой проход reference-модели (2) + обновление параметров (6) на токен
- Генерации GRPO входят в `C_train`, **не** учитываются повторно при развёртывании

**Обоснование k=10**: аналогично разложению `k=6` для SFT:
- Генерация completions: только forward pass → 2N на токен
- Reference model forward: → 2N на токен  
- Обновление policy (forward + backward): → 6N на токен
- Итого: 2 + 2 + 6 = 10

**Источник структуры**: Kaplan et al. [1], Snell et al. [4]. Конкретная формула для GRPO в литературе отсутствует; `k=10` — авторская оценка по аналогии с `k=6` для SFT.

---

### Inference (`k = 2`)

```
C_infer = 2 × N × T_gen × n
```

- `T_gen` — средняя длина генерации в токенах (из кэша генераций 04_inference)
- `n` — число генераций на задачу (1, 4, 8, 16, 32)
- `k = 2`: только forward pass
- KV-cache снижает реальное время, но не меняет оценку FLOPs по этой формуле

**Источник**: Snell et al. (2024), arXiv:2408.03314: *"for inference FLOPs, we use Y = 2ND_inference"*

---

### PRM-верификация (`k = 2`)

```
C_PRM = 2 × N_PRM × T_eval × n
N_PRM = 7 × 10⁹  (Qwen2.5-Math-PRM-7B)
T_eval ≈ T_gen
```

- Применяется только для стратегий `best_of_n` и `weighted_voting`
- `T_eval ≈ T_gen` — длина оцениваемой последовательности приблизительно равна длине генерации
- `k = 2`: только forward pass через PRM

**Источник**: по аналогии с inference; C_PRM = 2 × N_PRM × D_PRM

---

## Амортизация

```
C_total = C_train / M + C_infer + C_PRM
```

- `M` — число задач при развёртывании
- `C_train` амортизируется: обучение выполняется один раз, инференс — на каждую задачу
- Три режима: M = |D_test|, M = 10⁴, M = 10⁵

**Источник**: Wu et al. (2024), arXiv:2408.00724

## Допущения и ограничения

1. **Формула 6ND — верхняя оценка**: не учитывает attention (квадратичный по длине контекста). Поскольку все стратегии используют одно приближение, ранжирование на Парето-фронте сохраняется.

2. **LoRA не снижает FLOPs**: прямой и обратный проходы идут через все `N` параметров базовой модели.

3. **T_eval ≈ T_gen**: допущение для PRM. Реальная длина входа PRM может незначительно отличаться.

4. **k=10 для GRPO**: авторская оценка, внешних источников с верифицированной формулой FLOPs для GRPO не обнаружено. Структурно согласовано с k=6 для SFT.

5. **D_GRPO из лога**: `num_tokens` в TRL регистрирует токены completions policy-модели; вклад reference-модели и промптов в `num_tokens` не включён, но учтён через k=10.
