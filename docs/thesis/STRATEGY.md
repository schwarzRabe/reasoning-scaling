# Стратегия эксперимента: MVP → Этап 2

**ВКР МИФИ, МП — Колотий М.П.**  
**Тема:** Сравнение train-time vs inference-time compute для Qwen2.5-1.5B

---

## Центральный вопрос

> При фиксированном бюджете FLOPs — что выгоднее:  
> больше обучать или больше генерировать?

**Инструмент ответа:** Парето-фронт (FLOPs vs Accuracy) с точками пересечения стратегий.

## Два этапа

### MVP — минимально достаточный результат

**Цель:** защитимый Парето-фронт с имеющимися данными.

**Что входит:**
- 36 SFT-прогонов (обучение + eval на GSM8K и MATH-500)
- Inference для лучших SFT-моделей (majority voting, best-of-N, weighted voting)
- Парето-фронт при трёх режимах амортизации M
- Learning curve: accuracy vs объём данных

**Центральный результат MVP:**  
Точка первого бюджета, где inference превосходит SFT greedy — на GSM8K и MATH-500 раздельно.

**Что НЕ входит в MVP:**
- GRPO (только smoke test)
- Полный GRPO eval по всем чекпоинтам

---

### Этап 2 — расширение

**Цель:** добавить GRPO-ветку на Парето-фронт, проверить гипотезу о раннем crossover.

**Что добавляется:**
- GRPO на 4 лучших SFT-моделях (по val_loss)
- Eval каждые 50 шагов → точки на Парето-фронте
- Сравнение: SFT greedy vs GRPO greedy vs inference-стратегии
- Анализ при M = |D_test|, 10⁴, 10⁵

**Ключевая гипотеза:**  
Для Qwen2.5-1.5B точка пересечения стратегий наступает *раньше*, чем у крупных моделей (Snell et al., 2024).

## Методология FLOPs (кратко)

| Стратегия | Формула | D |
|-----------|---------|---|
| SFT | `6 × N × D` | токены датасета × эпохи |
| GRPO | `10 × N × D` | токены генераций из лога TRL |
| Inference | `2 × N × D` | T_gen × n |
| PRM | `2 × N_PRM × D` | T_eval × n |
| Итого | `C_train/M + C_infer + C_PRM` | амортизация по M |

Единый принцип `C = k × N × D` для всех стратегий.  
Источники: Kaplan et al. [1], Snell et al. [4].

## Ограничения 

1. **Разные протоколы eval:** SFT — greedy (temp=0), inference — sampling (temp=1.0)
2. **GRPO G=8** вместо G=64 из оригинала — ограничение VRAM
3. **Cross-model PRM:** верификатор обучен на Qwen-Math, генератор — на DeepSeek-R1
4. **T_gen из n=32** используется как прокси для всех n
5. **k=10 для GRPO** — авторская оценка, внешнего источника нет

## Практическая ценность

Руководство для инженеров, разворачивающих LLM на периферийных устройствах:

- Сколько примеров нужно для дистилляции CoT?
- При каком бюджете выгоднее генерировать несколько ответов вместо дообучения?
- Когда PRM-верификация окупает свою стоимость?
- Оправдывает ли себя GRPO при ограниченном VRAM?
